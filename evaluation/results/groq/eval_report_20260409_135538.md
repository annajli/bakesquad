## 4. Root Cause Analysis

### A. Prompt Design Assessment

The `query_plan_prompt` in `bakesquad/search/prompts.py` **already includes** the
correct examples:

```
Input: "banana bread"
→ {"category":"quick_bread", ...}

Input: "zucchini bread with chocolate chips"
→ {"category":"quick_bread", ...}
```

The schema definition also explicitly states:
> `quick_bread includes: banana bread, zucchini bread, pumpkin bread, muffins, scones, cornbread`

**Conclusion:** The prompt is *correctly designed*. If failures occur on these
specific queries, the root cause is **not** prompt omission — it is either:

1. **LLM reasoning inconsistency** — the model outputs correctly in most runs
   but fails on certain runs, especially with weaker models (Groq llama-3.1-8b).
2. **Model-specific knowledge gaps** — some models associate 'banana bread' with
   artisan/yeast bread rather than quick bread, overriding the in-context examples.
3. **Temperature/sampling variance** — at temperature=0 this should be deterministic,
   but quantised local models may not behave deterministically.

### B. Classification Schema Gaps

The schema has **four mutually exclusive, non-overlapping categories** with no
mechanism for partial or multi-label classification.

Gap items identified:
- `loaf` is not listed as a quick_bread synonym in the examples
- `bar cookies` (brownies, lemon bars) rely on the model knowing they are cookies
- No catch-all fallback within a category (e.g., 'if unsure between quick_bread/cake → quick_bread')

### C. Pipeline Rigidity

The pipeline is a **fixed 11-step DAG with no feedback loops:**

```
Step 1 → category (FIXED) → propagates to Steps 9, 10
Step 7 → may infer a different category from ingredients
        but this is IGNORED for ratio range selection
```

**Impact of a Step 1 misclassification:**
- Step 9 computes ratios against **wrong reference ranges** (e.g., cookie ranges for a quick bread)
- Step 10 uses **wrong scoring criteria** (e.g., brown/white sugar ratio for a loaf)
- No step corrects or revisits the category decision
- The final score is **meaningless** for misclassified recipes

## 5. Recommended Fixes

### Fix 1: Prompt Hardening (Low Effort, Immediate)

Add explicit `loaf` synonyms to the `query_plan_prompt` schema description:

```diff
- '  quick_bread includes: banana bread, zucchini bread, pumpkin bread, muffins, scones, cornbread\n'
+ '  quick_bread includes: banana bread, zucchini bread, pumpkin bread, muffins, scones, cornbread, '
+     'and ANY baked loaf that does not require yeast (banana nut loaf, zucchini loaf, etc.)\n'
```

Add a conflict-resolution tie-breaker example:

```
Input: "banana nut loaf"
→ {"category":"quick_bread", ...}   # loaf ≠ yeast bread
```

### Fix 2: Step 7 Category Reconciliation (Medium Effort)

After Step 7 (parsing), compare the parser-inferred category against the
Step 1 query plan category. If they disagree for a majority of fetched
recipes, override the plan category before Steps 9–10.

```python
# In run_pipeline(), after parse_recipes_parallel():
step7_categories = Counter(r.category for r in parsed if not r.parse_error)
dominant_step7 = step7_categories.most_common(1)[0][0] if step7_categories else None
if dominant_step7 and dominant_step7 != plan.category:
    logging.warning(
        'Category override: Step1=%s → Step7=%s (recipes=%s)',
        plan.category, dominant_step7, dict(step7_categories)
    )
    plan = plan.model_copy(update={'category': dominant_step7})
```

### Fix 3: Soft Classification (High Effort, Best Long-Term)

Replace the hard `Literal['cookie','quick_bread','cake','other']` with a
scored confidence dict, then select the category only when confidence is
above a threshold:

```python
# New QueryPlan field:
category_scores: dict[str, float]  # e.g. {"quick_bread": 0.9, "other": 0.1}
category: str  # argmax of category_scores

# Prompt change: ask the LLM for confidence scores instead of hard label.
# If max confidence < 0.6, log a warning and use the Step 7 re-classification.
```

### Fix 4: Add Regression Tests (Maintenance)

Add `pytest` tests that run the classification-only path (Stage 1) for the
five known-issue queries and assert `quick_bread`:

```python
# tests/test_classification.py
@pytest.mark.parametrize('query', [
    'banana bread', 'zucchini bread', 'chocolate chip banana bread',
    'moist zucchini bread with walnuts', 'gluten-free banana bread',
```

## 6. Architecture Evaluation

### Current Architecture: Fixed Sequential DAG

```
Query → [Step 1: classify] → [Steps 2-6: search+fetch] → [Step 7: parse]
      → [Steps 8-9: ratios] → [Step 10: score] → Output
         ↑ category locked here; no recovery downstream
```

**Strengths:**
- Deterministic, fast, predictable latency
- Minimal LLM calls (3–4 total per run)
- Easily debuggable (each step's output is a typed Pydantic model)

**Weaknesses:**
- Single point of failure at Step 1
- No correction loop if Step 1 is wrong
- Step 7 re-classification data is unused

### Alternative 1: Step 7 Feedback Loop

```
Query → [Step 1] → [Steps 2-6] → [Step 7] ─────────────────────┐
                                      ↓                         |
                              category disagrees?               |
                              yes → override plan.category      |
                              no  → continue as-is             |
                                      ↓                         |
                            [Steps 8-9] → [Step 10] ← ──────────┘
```

Cost: ~0 extra LLM calls. Recommended as Fix 2 above.

### Alternative 2: Soft Classification

Ask the LLM to return confidence scores for all categories.
Fall back to Step 7 when Step 1 confidence is low.
Cost: minor prompt change, same number of LLM calls.

### Alternative 3: Single-Pass Reasoning (ReAct-style)

Replace the entire pipeline with a ReAct loop that searches, reads pages,
and computes ratios as tool calls. More flexible but:
- Adds ~3–5× the LLM calls
- Unpredictable latency (would blow the 60 s budget)
- Harder to debug
- Not recommended given the tight budget constraints of this project

**Verdict:** Fix 2 (category reconciliation) gives the best improvement-to-effort
ratio. Fix 1 (prompt hardening) is a free win that should be done immediately.

## 7. Instrumented LLM Traces

All LLM prompts and raw responses are stored per-result in
`eval_results_<timestamp>.json` under the `llm_trace` key.

Each trace entry contains:
- `step_label` — which pipeline step made the call
- `system` — full system prompt
- `user` — user message
- `raw_response` — unmodified LLM output (before JSON extraction)
- `elapsed_s` — latency for this call
- `error` — exception message if the call failed

Use `--debug` flag to print traces to stdout during evaluation.
